# 測試案例撰寫規範（企劃書 → QA TestCase）

用途：拿到企劃書時，依本規範產出該功能的 QA TestCase xlsx，供人審、驗收與後續 AI/腳本自動化分流。

## 1. 產出檔案格式

- 檔名：`<企劃書檔名>_TestCase.xlsx`，一份企劃書一檔。
- 工作表：主表以系統名命名；大系統可拆多張表，但預設先產出一張主表。
- 主表欄位固定沿用團隊既有 TestCase 慣例：

| 欄 | 名稱 | 說明 |
|---|---|---|
| A | 項目 | 功能區塊，同區塊只在第一列填 |
| B | 類型 | `顯示確認` / `操作確認` / `數值確認` |
| C | TC內容 | 單一驗證點的一句話 |
| D | PASS/FAIL | 執行結果，留空待測 |
| E | Bug Key | 對應 bug 單號 |
| F | 備註 | 補充測試帳號需求、版本限制等 |
| G~K | 統計 | `PASS / FAIL / N/A / Total / TC數量` 公式統計 |

- 附屬工作表：
  - **優先度與自動化**：每條案例的 `P0~P2` 優先度與 `A/S/M` 自動化標記。
  - **待釐清問題**：規格模糊、寫不出對應 TC 的規格句清單。

## 2. 產出流程

1. 通讀企劃書，列出功能點清單：每個介面、按鈕、規則、數值公式、狀態與限制。
2. 對每個功能點跑一遍「覆蓋維度檢查表」，能套用的維度各寫至少一條。
3. 企劃書中每一句規格描述至少要有一條 TC 對應；寫不出對應 TC 的規格句列入「待釐清問題」。
4. 標上類型、優先度、自動化標記，並依項目分組排序：介面顯示 → 基本操作 → 數值 → 邊界異常。
5. 產出 xlsx，並在附屬頁保留企劃書版本、產出日期、待釐清問題。

## 3. 類型定義

| 類型 | 驗證什麼 | 例 |
|---|---|---|
| 顯示確認 | 靜態呈現：元素齊全、文案、數值、排序、狀態外觀、無缺破圖 | `商店介面正確顯示商品名稱、價格、剩餘次數` |
| 操作確認 | 互動行為：點擊/拖曳後的反應、跳轉、開關、流程 | `點擊『購買』鈕，正確跳出二次確認視窗` |
| 數值確認 | 數值變化與計算：扣款、增減、上限、重置、公式 | `購買成功後，金幣正確扣除商品標價金額` |

## 4. TC 內容句型

標準句型：

```text
[前置狀態/位置，] [操作，] 正確<預期結果>
```

- 顯示確認：`<介面/元素>正確顯示<內容>`；`<條件>時，<元素>正確<狀態>`。
- 操作確認：`點擊『<元素>』鈕，正確<結果>`；`在<介面>下<操作>，正確<結果>`。
- 數值確認：`<動作>後，<數值>正確<變化>（依據：標價/公式/上限）`。

寫作規則：

1. UI 元素一律用 `『』` 標注。
2. 前置條件寫進句首，例如 `玩家無公會時`、`貨幣不足時`。
3. 反向案例要寫出不可點、壓灰、不顯示、不會繼續、無法重複等可觀察結果。
4. 每個新介面至少一條美術完整性案例：`<介面>顯示正確無缺破圖`。
5. 排序與列表要指明規則，例如由大到小、由近到遠、依時間倒序。
6. 不使用「正常」「沒問題」「符合預期」等空泛詞，必須寫出具體預期。
7. 一條只驗一件事；複合驗證點要拆成多條。

### 操作動詞用語

| 手勢 | 標準動詞 | TC 範例 |
|---|---|---|
| 點一下 | 點擊 | `點擊『購買』鈕，正確跳出二次確認視窗` |
| 按住不放 | 長按 | `長按『道具』圖示，正確顯示道具詳情浮窗` |
| 按住移動物件 | 拖曳…至… | `將『角色』拖曳至上陣格，正確上陣` |
| 快速滑/瀏覽清單 | 滑動／捲動 | `在商品清單向上滑動，正確載入後續商品` |
| 雙指放大縮小/旋轉 | 雙指縮放／旋轉 | `雙指放大地圖，正確等比縮放不破圖` |

涉及雙指縮放/旋轉的 TC，因一般 ADB 腳本不穩定，自動化欄一律標 `M`。

## 5. 覆蓋維度檢查表

顯示層：

- 介面元素齊全：每個按鈕、圖示、文字欄位、頁籤各至少一條。
- 文案與數值顯示正確：價格、次數、倒數、等級需求。
- 各狀態外觀：選中/未選、可點/壓灰、已領/未領、紅點提示出現與消除。
- 空狀態：清單為空、無好友、無道具時顯示什麼。
- 滿/上限狀態：背包滿、次數用完、等級滿。
- 排序規則與無缺破圖。

操作層：

- 入口與返回：從哪些地方進入，`『X』` / `『返回』` 正確關閉回到上一層。
- 每顆按鈕功能、每個頁籤切換。
- 二次確認流程：取消不生效，確認才生效。
- 引導/教學：強制流程只能點指定處，可否跳過。
- 連點/重複操作：不會重複觸發或重複扣款。
- 操作中斷：關閉介面、切畫面、重開後狀態正確。

數值與經濟：

- 扣款/入帳金額 = 標價/公式，需記購買前後數值。
- 貨幣不足時擋下、提示、引導儲值。
- 次數限制：剩餘次數遞減、用完擋下、每日/每週重置。
- 獎勵發放：數量正確、不可重複領取。
- 上下限與邊界值：0、1、上限、上限+1。

條件與聯動：

- 開啟條件：等級、任務、前置系統未達成時鎖定與提示，達成後解鎖。
- 跨系統聯動：完成後任務進度、紅點、圖鑑、成就是否同步更新。

適配與異常：

- 多解析度下版面不跑版、不缺元素。
- 斷線、強退重進後狀態不遺失、不重複扣款。

## 6. 自動化欄位標記

- `A`：AI 白話測試可執行，適合截圖判讀的顯示、操作、數值案例。
- `S`：適合錄影腳本回歸，固定路徑、高頻重跑的核心流程。
- `M`：僅手動，包含手感、音效、動畫流暢度、多人互動、雙指操作、高風險消費流程。

## 7. 驗收標準

1. 企劃書每個規格句都有 TC 對應，或列入待釐清問題。
2. 每條 TC 單獨可判 PASS/FAIL，無複合驗證點、無模糊詞。
3. 每個新介面至少含：元素齊全 1 條、無缺破圖 1 條、入口/返回各 1 條。
4. 涉及貨幣/道具增減的功能，數值確認 P0 案例完整，含不足、上限、重置。
5. 統計欄公式正確，PASS/FAIL/N/A/Total 自動加總。

## 8. AI 輸出格式

只輸出以下純文字行，不要輸出 Markdown 表格或額外說明：

```text
SYSTEM: <系統名稱>
CASE: <項目>|<類型>|<優先度>|<自動化>|<TC內容>
ISSUE: <待釐清問題>
```

欄位限制：

- 類型只能使用：`顯示確認`、`操作確認`、`數值確認`
- 優先度只能使用：`P0`、`P1`、`P2`
- 自動化只能使用：`A`、`S`、`M`

## 9. 通用範例

| 類型 | 範例 |
|---|---|
| 顯示確認 | 商店介面正確顯示商品圖示、名稱、價格與剩餘次數 |
| 顯示確認 | 貨幣不足時，『購買』鈕正確壓灰不可點擊 |
| 顯示確認 | 商店介面顯示正確無缺破圖 |
| 操作確認 | 點擊『購買』鈕，正確跳出二次確認視窗 |
| 操作確認 | 二次確認視窗點擊『取消』鈕，正確關閉視窗且不扣款 |
| 操作確認 | 未完成前置任務時，點擊該系統入口，正確顯示未解鎖提示 |
| 數值確認 | 購買成功後，金幣正確扣除商品標價金額（購買前後數值相減=標價） |
| 數值確認 | 購買成功後，該商品剩餘次數正確 -1；次數歸 0 後『購買』鈕壓灰 |
| 數值確認 | 跨日後，每日限購次數正確重置為上限值 |
